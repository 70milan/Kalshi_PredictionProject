Decision 1 — Silver Storage Strategy
Chosen: Option B — Current State + History
Each data source gets two Silver tables:

A current state table — one row per entity, always latest, overwritten each run. This is what the dashboard and mispricing engine query.
A history table — append-only, every snapshot preserved. This is what Spark reads for pattern detection and model training.

Decision 6 — Gold Layer Design (plan only, build later)
The Gold layer joins all Silver sources around a Kalshi market to produce a mispricing score:
KALSHI MARKET (e.g. KXUSTARIFFSEU)
    │
    ├── Kalshi: odds moved > X% in last N snapshots?
    │
    ├── GDELT Events: NumMentions spike on related actors/countries?
    │   GoldsteinScale negative (destabilizing)?
    │
    ├── GDELT GKG: named persons matching market spiking?
    │   Relevant themes appearing?
    │
    └── NYT/Reuters: credible source covering same topic?
        Sentiment negative or conflict-oriented?

MISPRICING SCORE = weighted combination of all four signals
The actor-to-market matching logic (links GKG persons and GDELT events to Kalshi tickers) lives at Gold layer, not Silver. Silver just cleans. Gold joins and scores.

Architecture Reminder
This is a lakehouse architecture — not a data warehouse, not a traditional OLAP system:

Bronze/Silver/Gold on Delta Lake (Parquet + ACID)
DuckDB as SQL serving layer for ad-hoc queries and dashboard
PySpark for Silver/Gold transformations where GDELT volume justifies it
Medallion pattern, not dimensional modeling — Gold layer is analytical but not a formal star schema
